Recipes & Workflows,
explained simply.

A recipe is a one-click launcher. A workflow is the durable TypeScript program that runs when you click it. Together they let T3 Code do real, repeatable work without you writing a prompt every time.

Companion to docs/t3work-mvp/16-action-recipes.md, 25-workflow-engine.md, and 27-scheduled-workflows.md · generated Jun 2026

0The 30-second idea

Recipe

A launcher with a face. A small bundle of metadata + an icon + a card that says “Approve & merge PR”. Clicking it runs a workflow with sensible defaults and a short guided kickoff. You never type a prompt.

Workflow

A real TypeScript program that lives in a .workflow.ts file and runs under a durable, replay-based engine. It can call tools, spawn agent turns, ask you questions, wait for hours, and branch on errors — exactly like Temporal/Restate, but TS-native and journaled to SQLite.

The one-liner
A recipe is what you click. A workflow is what runs. The workflow can loop, wait, ask, and spawn agents — and when it does, each agent turn becomes a thread that shows up in the sidebar under whatever workitem it belongs to.

1Mental model

The whole system is built on four primitives (Epic 16): Context, Tools, Workflows, and Views. A recipe ties the first three together and offers itself on a Surface (a place in the UI).

Recipe Workflow Context Tools Surface / Engine

2The four primitives

Primitive What it is Owner
Context What the recipe/agent can see — the workitem, attached refs, profile, project. packages/project-context
Tools Typed, capability-gated actions: tools.jira.*, tools.github.*, scripts. SDK groups + tool broker
Workflows Durable TS programs (.workflow.ts) that orchestrate tools, agents, and asks. packages/t3work-sdk + server engine
Views JSX card templates that render the recipe / workflow messages in the UI. t3work-skill-packs + web

A recipe is metadata that binds these: it points at a defaultAction (a workflow ref), declares which Context it needs, on which Surfaces it appears, and which Views render its card and its workflow’s conversation messages.

3Recipes, end to end

3.1 · Anatomy of a recipe

Recipes live in three scopes. The two you’ll touch most:

  • project-local <managed-project>/.t3work/recipes/<id>/ — your team’s recipes, checked into the project.
  • bundled shipped in packages/t3work-skill-packs — the defaults that ship with T3 Code.

A project-local recipe is a small folder:

# .t3work/recipes/pr-approve-and-merge/
recipe.ts            # metadata + defaultAction + surfaces + appliesTo
approve.workflow.ts  # the durable program
views/               # JSX card templates (optional)
tools/               # recipe-scoped tools (optional)

The Recipe schema (UI-facing projection) is the contract the server uses to match recipes to surfaces and render cards: 

// packages/project-recipes/src/recipe.ts
export const Recipe = Schema.Struct({
  id: Schema.String,
  title: Schema.String,
  shortDescription: Schema.String,
  surfaces: Schema.Array(RecipeSurface),
  promptTemplate: Schema.optional(Schema.String),
  kickoff: Schema.optional(ProjectRecipeKickoffProgram),
  appliesTo: RecipeApplicability,
  requiredContext: Schema.Array(RecipeContextRequirement),
  outputPreference: RichOutputPreference,
  artifactKinds: Schema.optional(Schema.Array(Schema.String)),
  suggestedActions: Schema.optional(Schema.Array(RecipeFollowup)),
});

Authors don’t write that schema by hand. They use the typed SDK:

// .t3work/recipes/pr-approve-and-merge/recipe.ts
import { defineRecipe, defineWorkflow } from "@t3work/sdk";

export const approveMerge = defineRecipe({
  id: "pr-approve-and-merge",
  version: "1.0.0",
  scope: "project",            // only project-scoped recipes are supported
  title: "Approve & merge PR",
  shortDescription: "Approve, then merge — escalate on protected-branch errors.",
  surfaces: ["github.pull_request.detail.sidepanel"],
  defaultAction: defineWorkflow("./approve.workflow.ts"),
});

3.2 · Surfaces & discovery

A surface is just a string literal — a place in the UI where a recipe card can appear: 

// packages/project-recipes/src/surface.ts
export const RecipeSurface = Schema.Literals([
  "project.dashboard.backlog",
  "project.dashboard.myWork",
  "workitem.detail.sidepanel",
  "thread.context",
  "github.pull_request.detail.sidepanel",
  "github.pull_request.diff.selection",
  "github.review.comment",
]);

Discovery is server-side and visibility-aware. When the web asks for recipes for a given render context, the server:

  1. Scans .t3work/recipes/* (project-local) and merges in bundled recipes.
  2. Filters by surfaces — only recipes declaring the current surface survive.
  3. Evaluates appliesTo against the render context (workitem type, integrations connected, profile).
  4. Ranks survivors via matchRecipes + profile weights.

3.3 · “Save as recipe” — the offer, never silent

Constitution rule (AGENTS.md)
After completing a repeatable t3work workflow, agents should mention that it could be saved as a project-scoped action recipe and offer to create it. Do not create recipes silently. The bundled create-recipe meta-recipe scaffolds the folder via an explicit user confirmation.

This is also the user-facing framing in Epic 28: don’t say “I created a workflow” — say “I can set this up so it’s one click next time.”

4Workflows, end to end

4.1 · Durable & replay-based

A workflow is a real TypeScript body in a .workflow.ts file. There is no defineWorkflow(async (ctx) => …) wrapper — the body is the function. The engine journals every primitive call and replays the body on resume to reach the next live call. Suspend for an hour, restart the server, come back — the run continues from the journal. 

// .t3work/recipes/pr-approve-and-merge/approve.workflow.ts
import { Schema } from "effect";
import { githubWrite } from "@t3work/sdk/groups";

export const Inputs = Schema.Struct({ prId: Schema.String });
export const Outputs = Schema.Union(
  Schema.Struct({ status: Schema.Literal("merged"), sha: Schema.String }),
  Schema.Struct({ status: Schema.Literal("blocked"), reason: Schema.String }),
);

export const meta = {
  name: "pr-review.approve-and-merge",
  capabilities: ["user", githubWrite],
  phases: [{ title: "Approve" }, { title: "Merge" }] as const,
} as const;

const input = Schema.decodeSync(Inputs)(args);

phase("Approve");
await tools.github.pullRequest.approve({ id: input.prId });

phase("Merge");
try {
  const { sha } = await tools.github.pullRequest.merge({ id: input.prId });
  return { status: "merged", sha };
} catch (e) {
  if (e instanceof PermissionDeniedError) {
    const ask = await user.ask({
      title: "Branch protected — request admin override?",
      responseSchema: Schema.Struct({ proceed: Schema.Boolean }),
    });
    const decision = await ask.response;
    if (!decision.proceed) return { status: "blocked", reason: "declined" };
    const { sha } = await tools.github.pullRequest.merge({ id: input.prId, adminOverride: true });
    return { status: "merged", sha };
  }
  throw e;
}

4.2 · Workflow primitives (the globals)

Inside a .workflow.ts, these are ambient globals injected by the engine:

Primitive What it does Suspends on?
tools.* Typed, capability-gated external calls (jira, github, scripts). No (unless it returns a Handle)
agent() / askAgent() Run an LLM turn, optionally in a child thread. Yes — until turn completes
user.ask() Render a typed form/card in the UI and wait for the reply. Yes — until user replies
thread.notifyUser() Post a rich message; fire-and-forget. No
spawnThread() Create a child thread and get a Handle to drive it. Depends on call
waitUntil(date) Sleep until a wall-clock deadline (Epic 27). Yes — until clock
phase(title) Mark progress (pure metadata, journaled). No
now() The journaled clock — replay-safe time read. No
Where it runs
Server: t3work-workflowEngineLaunch.ts (launch + journal + DB lifecycle), t3work-workflowEngineReactor.ts (wakes askUser/askAgent on events), t3work-workflowScheduler.ts (clock wake for waitUntil), t3work-workflowEngineRegistry.ts (in-memory parked-run index). Persistence: workflow_runs + workflow_journal (SQLite).

5Routines & the clock

Epic 27 adds one primitive (waitUntil) and one service (the scheduler). A routine is just a workflow that loops on a timer. There is no “automations panel.” A routine is a thread that lives in the sidebar, mostly asleep, waking on its timer to post what it did and sleep again.

6Threads in the sidebar

Every workflow runs inside a thread. Threads are grouped in the project sidebar, and crucially: a thread can be attached to a workitem (a Jira ticket) via ticketId, and a child thread can be nested under a parent via parentThreadId. That’s how agent handoffs show up under the kickoff that spawned them.

The grouping logic lives in t3work-projectSidebarThreadGroups.ts and the parent/child tree in t3work-projectSidebarThreadTree.ts. The ProjectThread type carries the linkage fields: 

// apps/web/src/t3work/t3work-types.ts
export type ProjectThread = {
  id: string;
  projectId: string;
  parentThreadId?: string;     // nested handoff
  ticketId?: string;           // attached workitem
  dashboardMode?: "backlog" | "my-work";
  displayMode?: "embedded" | "thread";
  sleepingUntil?: string;      // scheduled wake (Epic 27)
  status: "idle" | "running" | "completed" | "error";
};

7Kanban today — and what’s missing

T3 Code already renders a kanban board from Jira tickets. The lane a ticket lands in is derived from its Jira status string by keyword matching: 

// apps/web/src/t3work/t3work-projectTicketStatus.ts
export function getProjectTicketKanbanLane(status: string)
  : "todo" | "inProgress" | "review" | "done" {
  const s = normalizeProjectTicketStatus(status);
  if (s.length === 0) return "todo";
  if (includesStatusKeyword(s, doneStatusKeywords))      return "done";
  if (includesStatusKeyword(s, reviewStatusKeywords))    return "review";
  if (includesStatusKeyword(s, todoStatusKeywords))      return "todo";
  return includesStatusKeyword(s, inProgressStatusKeywords) ? "inProgress" : "todo";
}
The gap
There is no dedicated workitem state-machine module today — Jira is the source of truth for status, and T3 Code only mirrors it. There’s also no concept of “this lane means: an agent should pick it up automatically.” The board is a view; nothing watches it.

Everything we need to build that watcher already exists, though:

Capability Status Where
Workflow that loops + sleeps shipped Epic 27 — waitUntil + scheduler
Poll Jira for changes shipped t3work-integration-polling.ts + atlassian routes
Spawn an agent in a child thread shipped spawnThread + tool broker handoff
Create isolated worktree + feature branch shipped t3work-toolBrokerStartChildContext.ts + GitWorkflowService
Nest child thread under a workitem shipped ticketId + parentThreadId + sidebar tree
A “ready for agent” lane / state missing Needs a project-local convention + a watcher

8Use case: a kanban lane that auto-runs agents

Here’s a concrete, end-to-end design for a workflow that turns a kanban column into an auto-agent lane: drag a ticket to Ready for Agent, and an agent spins up in an isolated worktree + feature branch, attached as a new child thread under that workitem in the sidebar.

8.1 · Design

  • State cycle. Add one project-local lane convention: a Jira status whose name contains Ready for Agent (or a configured status id). The existing lane-mapper already buckets by keyword — we just add a new keyword bucket and treat it as a trigger, not a view-only lane.
  • Watcher. A routine (Epic 27 looping workflow) per project that polls the backlog for tickets in the ready-for-agent lane. Polling uses the existing fingerprint envelope so unchanged boards cost almost nothing.
  • Kickoff. For each ready ticket, the routine spawns a child agent thread attached to the workitem (ticketId) and moves the ticket to In Progress (Agent) so the same ticket isn’t picked twice.
  • Isolation. The agent runs in a fresh worktree under .t3work/child-session-worktrees/<ticket>-<short> on a feature branch t3work/<ticket>/<slug>. This is exactly what t3work-toolBrokerStartChildContext.ts already does for child handoffs.
  • Sidebar. Because the child thread is created with ticketId + parentThreadId, it shows up nested under the workitem in the sidebar — exactly like a manual kickoff would.
  • Termination. On agent completion, the workflow posts a rich result card, opens (or updates) a PR from the feature branch, and moves the ticket to Review. On failure, it moves the ticket back to Ready for Agent (or Blocked) with a diagnostic artifact.

8.2 · The flow

8.3 · The workflow sketch

This is a single .workflow.ts plus a tiny recipe.ts. It reuses every primitive above — no new engine concepts. 

// .t3work/recipes/ready-agent-watcher/recipe.ts
import { defineRecipe, defineWorkflow } from "@t3work/sdk";

export const readyAgentWatcher = defineRecipe({
  id: "ready-agent-watcher",
  version: "1.0.0",
  scope: "project",
  title: "Auto-agent lane watcher",
  shortDescription: "Watch the ‘Ready for Agent’ lane and spin up an isolated agent per ticket.",
  surfaces: ["project.dashboard.backlog"],
  defaultAction: defineWorkflow("./watcher.workflow.ts"),
});
// .t3work/recipes/ready-agent-watcher/watcher.workflow.ts
import { Schema } from "effect";
import { jiraRead, jiraWrite, githubWrite } from "@t3work/sdk/groups";
import { nextDelay } from "@t3work/sdk/time";

export const Inputs = Schema.Struct({
  jqlReady: Schema.String.payload(`status = "Ready for Agent"`),
  jqlClaimed: Schema.String.payload(`status = "In Progress (Agent)"`),
  pollSeconds: Schema.Number.payload(60),
});

export const meta = {
  name: "ready-agent-watcher",
  capabilities: ["schedule", "user", jiraRead, jiraWrite, githubWrite],
  phases: [{ title: "Wait" }, { title: "Scan" }, { title: "Claim" }, { title: "Run" }, { title: "Finish" }] as const,
} as const;

const input = Schema.decodeSync(Inputs)(args);
const ReadyTicket = Schema.Struct({
  id: Schema.String, key: Schema.String, summary: Schema.String,
});

while (true) {
  phase("Wait");
  await waitUntil(nextDelay(now(), { seconds: input.pollSeconds }));

  phase("Scan");
  const ready = await tools.jira.searchIssues({ jql: input.jqlReady });
  if (ready.items.length === 0) continue;

  for (const item of ready.items) {
    const ticket = Schema.decodeSync(ReadyTicket)(item);

    phase("Claim");
    // Atomic claim: move to "In Progress (Agent)" before doing anything.
    await tools.jira.transitionIssue({ id: ticket.id, status: "In Progress (Agent)" });

    phase("Run");
    // spawnThread with ticketId + parentThreadId → nested under workitem in sidebar.
    const run = spawnThread({
      name: `${ticket.key} agent`,
      ticketId: ticket.key,
      parentThreadId: thread.id,
      // Tool-broker child context creates the worktree + feature branch.
      childContext: { kind: "worktree", baseRef: "main" },
    });

    const result = await run.askAgent(
      `Implement ${ticket.summary} (${ticket.key}). ` +
      `Work on your feature branch, commit in logical chunks, push, and open a PR back to main. ` +
      `Post a short progress note after each commit.`,
      { attachments: [item] },
    );

    phase("Finish");
    if (result.status === "completed") {
      await tools.jira.transitionIssue({ id: ticket.id, status: "Review" });
      await thread.notifyUser(`${ticket.key} → Review. PR opened.`, { attachments: [result.prRef] });
    } else {
      await tools.jira.transitionIssue({ id: ticket.id, status: "Ready for Agent" });
      await thread.notifyUser(`${ticket.key} agent failed — requeued.`, { attachments: [result.diagnosticArtifact] });
    }
  }
}
Why this fits cleanly
Every moving part already exists:
  • waitUntil + loop = the routine (Epic 27).
  • tools.jira.searchIssues + fingerprint poll = existing polling.
  • spawnThread({ ticketId, parentThreadId, childContext: { kind: "worktree" } }) = existing child-handoff + worktree creation in t3work-toolBrokerStartChildContext.ts.
  • Sidebar nesting under the workitem = existing ProjectThread + tree builder.
  • Ticket lane transitions = existing transitionIssue tool + lane mapper.
The only genuinely new bits are: a status-keyword convention for the “Ready for Agent” lane, and a claim transition to prevent double-pickup.

Each ticket it picks up gets its own child thread nested under the ticket card — so from the user’s view, the agent work is discoverable exactly where they’d expect: under the workitem, not buried in some “automations” panel (which Epic 27 explicitly rejects).

9One-screen summary

Recipes

  • Launcher + metadata + card on a surface.
  • Project-local under .t3work/recipes/*, or bundled.
  • Point at a defaultAction workflow.
  • Offered (never silently created) after repeatable runs.

Workflows

  • Real TS in .workflow.ts, no wrapper.
  • Durable: journal + replay, suspend/resume.
  • Primitives: tools, agent, askUser, waitUntil, spawnThread.
  • Run inside a thread; thread attaches to a workitem.

Routines

  • A workflow that loops on waitUntil.
  • No automations panel — it’s a sleeping sidebar thread.
  • Scheduler wakes it on the wall-clock deadline.

Auto-agent lane (use case)

  • One new status convention: Ready for Agent.
  • A routine polls, claims, spawns an agent in a worktree + branch.
  • Child thread nested under the workitem in the sidebar.
  • Drives the ticket forward: Ready → In Progress (Agent) → Review.
