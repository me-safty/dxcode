Native Provider Tool Injection Plan

Goal: inject t3work custom tools into each provider using the most native runtime path available, with strict fallback rules and no provider-specific business logic leakage.

Native SDK Injection First MCP as Structured Fallback File Discovery Last Resort Adapter Isolation + Durable Artifacts

North Star Decisions

Product Principle

The user experience should feel provider-native, not prompt-hacked. Commands, tool calls, user input, and rich outputs should flow through native provider semantics whenever possible.

Architecture Principle

One shared t3work tool contract, multiple thin provider adapters. Keep workflow logic in shared packages; keep provider-specific code at transport and registration boundaries only.

Injection Strategy Ladder

Step 1: Runtime Native Injection

Preferred. Register tools through provider SDK/session APIs during startup or live session updates. Use this whenever supported.

Step 2: Native MCP Registration

Use provider-supported MCP flows if runtime tool registration is not supported. Keep setup transparent and diagnostics visible.

Step 3: Discovery Fallback

Last resort only. Use filesystem/discovery conventions behind adapter boundaries with a migration target to Step 1 or Step 2.

Must-Capability Contract (v0)

Host-Agent Bridge

Every tool call must run through host backend authority with correlated lifecycle states and idempotent invocation IDs to prevent duplicate side effects.

Context-Scoped Execution

Every invocation must carry explicit scope (tenant/account/project/thread/actor) and optional app-state snapshots; no ambient global access is allowed.

Security Boundaries

Deny-by-default outside current scope, enforce server-side authorization, require approval gates for mutation tools, and redact secrets in outputs and logs.

Rich Rendering Flexibility

Support predefined app-style UI payloads and generated HTML payloads, each renderable as conversation cards or persistent sidecar views with versioned refresh semantics.

Mutation Control Rule (Hard Gate)

What Agent Can Do

Prepare mutation drafts, call validation/read endpoints, and continuously refresh card/sidecar state so users see live previews while the plan evolves.

What Agent Cannot Do

Trigger final external commit actions directly. Save/apply/delete/create transitions must only happen from explicit user clicks in the rendered card or persistent sidecar UI.

Required Commit Gate

Backend must require user-click provenance (short-lived action token bound to scope + invocation) before executing any external mutation, even when the agent initiated draft generation.

Computer Use Tech Decision

Decision: separate Browser Use and Computer Use as two distinct tracks. Playwright is the Browser Use default. Computer Use evaluates two primary candidates in parallel: CUA-style runtime (for example trycua/cua) and xa11y-style accessibility-tree-first runtime.

Option Scope Strengths Risks / Gaps Fit For t3work Contract Decision
Playwright (OSS) Browser Use Mature cross-browser automation, strong TypeScript DX, robust trace/screenshot/video capture Browser-centric only; not full desktop computer control Excellent for deterministic step logs, durable evidence, and commit gates Browser Use default
CUA (for example trycua/cua) Computer Use Built for broader computer interaction beyond DOM-only flows Less deterministic than DOM automation; needs stronger checkpoint policy Strong when host keeps execution authority and click-gated irreversible steps Primary candidate
xa11y-style runtime Computer Use Accessibility-tree-first actions can improve reproducibility in supported apps Depends on target accessibility quality and platform support Strong for guarded enterprise workflows when accessibility surfaces are reliable Primary candidate
Puppeteer Browser Use Simple Chromium automation with broad ecosystem Weaker cross-browser strategy and fewer built-in testing ergonomics Good for Chromium-only flows; weaker for provider parity Not primary
Selenium / WebDriver BiDi Browser Use Enterprise familiarity and broad compatibility Higher protocol and infrastructure complexity for MVP Acceptable for compatibility cases but slower iteration loop Defer
Managed Playwright (for example Browserbase) Browser Use Elastic hosted sessions and easier remote execution Cost/vendor dependency and governance review requirements Strong if wrapped by the same adapter and artifact schema Optional scale mode
Implementation Guardrails
  • Host backend owns execution through two contracts: BrowserUseAdapter and ComputerUseAdapter.
  • Browser Use uses DOM-centric actions (navigate, click, type, select, waitFor, snapshotDom).
  • Computer Use uses computer-centric actions (observeScreen, movePointer, clickAt, typeText, hotkey, requestCheckpoint).
  • Every step emits durable artifacts (screenshot + context summary + action log).
  • Terminal side effects (submit/create/delete/transition) require user-click provenance token validation.
  • Any managed runtime must preserve invocation, scope, and artifact semantics.

Implementation Blueprint

1) Shared Tool Contract

Define typed manifest and execution contract for all t3work tools: ids, input/output schemas, auth scope, side-effect class, artifact policy.

2) Provider Adapters

Adapter responsibilities: capability probe, strategy selection, native registration, diagnostics, lifecycle event mapping.

3) Canonical Event Bridge

Normalize tool lifecycle, user-input questions, and plan updates into existing orchestration activity flow so current t3code UX works without forked logic.

Phase Plan

Phase 0: Capability Evidence

Fill matrix with direct SDK/runtime evidence for each provider and lock preferred strategy.

Phase 1: Contract + Adapter Interface

Create shared t3work tool contract and provider adapter interface with test fixtures.

Phase 2: First Native Provider Pilot

Implement one provider end-to-end with runtime-native injection and canonical event mapping.

Phase 3: Multi-Provider Rollout

Add remaining providers using same contract, then only fallback where strictly necessary.

Phase 4: Rich Tool UI

Ship sidecar-style card artifacts and interactive workflows for scoped Jira and mutation previews.

Design Risks to Watch

Risk: Hidden Provider Drift

If provider capability assumptions are wrong, fallback usage can silently expand. Mitigation: explicit capability probes + runtime diagnostics surfaces.

Risk: Business Logic Duplication

Re-implementing workflows per provider increases maintenance load. Mitigation: strict adapter boundary and single shared tool execution core.
