Implementation plan

Replace the pathwayOS Connect Relay with Convex

A staged plan to move the hosted relay control plane from the current Cloudflare Worker, PlanetScale, Alchemy, Hyperdrive, and queue stack to Convex while keeping Clerk as identity and making Cloudflare Tunnel/DNS an explicit remote-connection transport only.

Convex backend Clerk identity Remote connection provider boundary Cloudflare only on demand APNs later phase

Executive Summary

Yes, this can be done. The safest version is not a direct rewrite of every relay file. It is a product and architecture split: Convex becomes the account and Connect control plane, while Cloudflare Tunnel/DNS becomes a pluggable remote-access provider that is only used when a user deliberately creates or enables a remote connection.

The current relay is doing three separate jobs that should not all be tied to login: account identity, account/device/environment state, and remote network exposure. The Convex migration should preserve that separation. Signing in should never require Cloudflare configuration, tunnel provisioning, DNS records, APNs, or a local cloudflared runtime.

The new model: Clerk authenticates the user, Convex stores and syncs account-level state, and remote access providers are activated only by explicit user intent from a Connections or Remote Access flow.

Login and sync stop at Clerk plus Convex. Remote exposure starts only after explicit user action.

Always-on account layer

Clerk and Convex support login, profile, user preferences, device records, workspace sync, and non-network-exposing account features. These paths must work without Cloudflare credentials, DNS zones, APNs keys, or tunnel binaries.

On-demand remote layer

Cloudflare Tunnel/DNS becomes a transport provider selected during remote connection creation. This provider can be unavailable, disabled, or swapped later without blocking login or general app sync.

Core Product Rule

Cloudflare Tunnel/DNS must be tied only to explicit remote-connection workflows. It must not be part of sign-in, profile rendering, account sync, local pairing, settings load, or app startup.

Concrete gating rules

  • Signed-out users go to Clerk login; no Convex Connect provider work is attempted.
  • Signed-in users can load profile, plan label, account preferences, and local app state without Cloudflare config.
  • Connections can show an empty remote state when no remote provider is configured.
  • Remote connection creation is the first moment the app asks for or validates Cloudflare provider readiness.
  • If Cloudflare is missing, show a remote-access setup state, not an auth failure.
  • Local server pairing remains independent from Clerk and Convex remote access.

Proposed Backend Package

Add a dedicated Convex backend workspace package rather than mixing Convex functions into the web app. Suggested package name: infra/connect-convex or apps/connect. The package owns Convex schema, HTTP actions, internal mutations, scheduled functions, and provider modules.

Implementation Phases

Phase 0: Architecture lock and test harness

  • Freeze the current relay external contract in tests using shared schemas.
  • Document which routes are account-only, remote-provider-only, and mobile-only.
  • Add fixture-based tests for link challenge, link proof, DPoP token exchange, replay rejection, and environment credential minting.
  • Define an explicit RemoteEndpointProvider interface with methods for provision, deprovision, status metadata, and runtime config.

Phase 1: Scaffold Convex backend

  • Add a dedicated Convex workspace package and convex.json.
  • Configure Node actions for crypto/APNs/provider calls.
  • Add Convex schema, indexes, generated API, and package scripts for dev, deploy, codegen, test, and typecheck.
  • Wire CI to typecheck Convex functions without deploying production.

Phase 2: Clerk and account baseline

  • Configure Clerk auth in Convex and validate the same Clerk user id contract used by the app.
  • Add account bootstrap, profile metadata sync, and Free plan fallback in Convex.
  • Update web app configuration so login/profile requires only Clerk and Convex, not remote-provider settings.
  • Add tests proving missing Cloudflare config does not block login, profile, settings, or account sync.

Phase 3: Port relay-compatible HTTP Actions

  • Implement /health, metadata, list environments, list devices, link challenge, link environment, unlink, status, connect, mobile registration, and publish activity routes.
  • Keep route response shapes compatible with @pathwayos/contracts.
  • Map Convex errors into existing relay-style error codes with support IDs.
  • Keep the existing client runtime pointed at PATHWAYOS_RELAY_URL while swapping the URL to Convex.

Phase 4: Port token and proof logic

  • Move EdDSA JWT signing and verification helpers into a shared package usable by Convex Node actions.
  • Port link challenge signing, DPoP access token issuing, DPoP proof verification, replay guard persistence, cloud mint proof signing, and environment response verification.
  • Use Convex mutations for atomic replay guard creation.
  • Add golden tests against the current relay to prove token claims and rejection behavior match.

Phase 5: Decouple Cloudflare into remote provider

  • Rename product state from generic cloud enabled to remote access provider readiness where appropriate.
  • Move Cloudflare credentials out of global relay requirements and into provider-specific Convex env vars.
  • Do not provision tunnels during login, profile load, device sync, environment listing, or ordinary environment link challenge creation.
  • Provision Cloudflare Tunnel/DNS only when the user chooses Create Remote Connection or Enable Remote Access.
  • Store provider allocation status separately from environment link status so a linked environment can exist without remote exposure.
  • Expose provider setup errors as Connections UI states, not auth errors.

Phase 6: Update local server and CLI

  • Preserve local server endpoints for link proof, relay config, health, and mint credential.
  • Update CLI pathwayos connect commands to call Convex HTTP Actions or Convex client actions.
  • Keep desired-link state local, but distinguish linked account from remote exposure intent.
  • Make tunnel runtime startup depend on stored provider runtime config only, not generic account sign-in.

Phase 7: Mobile and APNs

  • Port mobile device registration and Live Activity token registration.
  • Port activity aggregation and delivery scheduling.
  • Replace Cloudflare Queue semantics with a Convex delivery table and scheduled retry worker.
  • Keep APNs credentials optional until mobile notifications are enabled.

Phase 8: Cutover and decommission

  • Run Convex and the existing relay side by side behind separate URLs.
  • Ship a feature flag for Convex Connect backend selection.
  • Validate new links on Convex first; leave existing relay links untouched until migration tooling exists.
  • Build a migration script only after the Convex data model survives real link/connect tests.
  • Remove Alchemy, PlanetScale, Hyperdrive, and Cloudflare Worker deploy once no clients point at the old relay.

Remote Connection UX Changes

Before

  • Cloud config is treated as a broad account capability.
  • Missing relay/provider config can make account surfaces look unavailable.
  • Environment linking and remote exposure are closely coupled.

After

  • Account sign-in and sync are Clerk plus Convex only.
  • Connections shows local link state, account state, and remote exposure state separately.
  • Cloudflare appears only inside a Remote Access provider setup or create-connection flow.

Suggested UI states

  • Signed out: show sign-in CTA only.
  • Signed in, no remote provider: account sync works; Connections shows "Remote access not configured".
  • Provider configured, no environment exposed: show "Create remote connection".
  • Provisioning: show tunnel/DNS progress and recoverable provider errors.
  • Remote ready: show endpoint status, connect action, and unlink/deprovision controls.

Testing and Verification

  • Run existing repo checks: vp check and vp run typecheck.
  • Add Convex function tests for schema, auth helper, link proof validation, replay rejection, and provider allocation state transitions.
  • Add web tests showing login/profile/settings work without Cloudflare provider env vars.
  • Add Connections tests for provider unavailable, provider configured, provisioning failed, remote ready, unlink, and retry states.
  • Add integration tests for full link and connect against a fake provider before calling Cloudflare APIs.
  • Add one manual end-to-end smoke test with a real Cloudflare-managed remote endpoint before production cutover.
  • Add APNs delivery tests with transport mocked at the APNs boundary.
  • Validate desktop and browser flows separately because Clerk/Electron token persistence differs from browser Clerk.

Risks and Mitigations

Risk Mitigation
Security drift in DPoP or JWT claims Golden tests against current relay behavior before client cutover.
Provider provisioning side effects inside Convex actions Use idempotent provider operations and explicit allocation state documents.
Convex HTTP Action limits or runtime differences Keep HTTP actions thin; move crypto/APNs/provider calls to Node actions where needed.
Custom domain delays Start with Convex site URL; add custom domain later as a deployment polish step.
Existing relay-linked environments Do not auto-migrate on day one. New links use Convex, old links stay on old relay until migration tooling is proven.
Cloudflare still feels required Audit every auth/account path for provider calls and add tests that run with provider config absent.

Decision Points

  • Should Convex replace only the relay API, or also become the source of truth for broader account/workspace sync?
  • Should remote access support manual endpoints alongside Cloudflare-managed endpoints from v1?
  • Do we want to keep the old PATHWAYOS_RELAY_URL name for compatibility, or rename to PATHWAYOS_CONNECT_URL with a compatibility alias?
  • Do we require a custom domain for production cutover, or is a Convex site URL acceptable for the first private beta?
  • Should APNs move in the first Convex release, or follow after remote connect is stable?
