Executive Direction

Keep

Existing app access model

The current app has a server/pairing gate for local and hosted environment access. Do not replace that with Clerk. Clerk should own cloud identity, profile, managed relay tokens, and future billing/account state.

Change

Make Clerk identity visible everywhere

Move the visible account affordance out of Settings and into the main sidebar footer. Settings stays available as a menu action inside the profile area, plus via the command palette and app menu.

Implementation Phases

Phase 1

Auth route foundation

Add the route files and shared auth shell. Wire Clerk route URLs in ClerkProvider and ElectronClerkProvider. Redirect signed-in users away from login/register. Show a clear unavailable state when cloud public config is absent.

Phase 2

Profile footer replacement

Replace the Settings button in SidebarChromeFooter with a profile area. Signed in: avatar, email, plan/status line, mobile-client indicator, and menu actions. Signed out: compact "Sign in" CTA that opens /login. Keep Settings reachable as an item in the profile menu.

Phase 3

Profile menu content

Extend the Clerk UserButton with existing "Mobile clients", a Settings link, and later plan/billing links. The first version should show Free as the temporary tier. When billing permissions are built, replace the fallback with the user's current Clerk Billing plan.

Phase 4

Password recovery polish

Verify Clerk's installed package behavior with path routing. Use the prebuilt reset flow where possible. If product requires a distinct screen, implement a custom useSignIn flow behind the same shell and test email-code reset, invalid code, resend, and successful redirect.

Phase 5

Docs, tests, and release readiness

Update Clerk setup docs and any onboarding copy that says the sidebar has no account control. Add route/shell tests, profile footer rendering tests, and regression coverage for missing cloud config.

Risks And Decisions

Decision

Where does the plan label come from?

The profile row should eventually display the user's current Clerk Billing plan. For this first pass, show Free and defer billing permissions, entitlements, and paid-plan display logic to a later billing phase.

Risk

Two auth models

The server pairing gate protects local environment access. Clerk protects cloud account identity. The UI must make this feel coherent without weakening either boundary.

Risk

Electron routing

Electron uses hash history. Clerk path routing should be verified in both browser and Electron shells, especially reset-password and callback/task URLs.
